# Product Recommendation Demo

An end-to-end demo of a **retail-robot product recommender**: it categorizes the store's
products, tags each with attributes, and runs a **multi-turn conversation state machine**
that infers what the shopper needs, asks clarifying questions when unsure, recommends
in-stock SKUs, **remembers preferences and builds a cart across the whole conversation**,
and finishes with a **checkout + robot pick-plan**. A **deterministic rule engine answers every
turn instantly, fully offline**; when a message falls outside what the rules recognise, that turn is
handed to a **real LLM running in the visitor's own browser** (WebLLM · WebGPU) — **no server, no
API key, no install**. Bilingual **EN / 中文**.

Built around the real inventory in [`../grasp-config/goods_prior_config.yaml`](../grasp-config/goods_prior_config.yaml).

## Multi-turn capabilities

The conversation persists — it does **not** reset after each recommendation.

- **Session memory** — a stated preference (*"I don't like sweet drinks"*, *"no caffeine"*)
  becomes a **standing constraint** applied to every later need. Toggle with the *memory*
  selector in the header; learned prefs show in the **Session memory** panel.
- **Cart** — accepting an item adds it to a running cart; the bot then asks *"anything
  else?"*. Add more items, `remove` one, or say *"that's all / check me out"*.
- **Quantity** — *"make it two"*, *"three of them"*, *"20"* (auto-capped to stock).
- **Anaphora / references** — *"the pink one instead"*, *"the second one"*, *"what else do
  you have?"* → the bot lists options you can pick by ordinal or by name.
- **Refine loop** — *"too sweet"*, *"something colder"*, *"no caffeine"* re-rank without ever
  re-offering a rejected item.
- **Checkout** — summarizes the cart and emits the **robot action plan** (turn direction +
  pick count per item, with the `pick_ice_cream` BT flagged), then *"please wait a moment."*

## The three parts (as specified)

1. **Categorize products + assign attributes (model-generated).**
   [`catalog.json`](catalog.json) — 20 in-stock SKUs, each with a `category`, `function_tags`
   (from a fixed 5-tag vocabulary), `sweetness` 0–5, `sugar_level`, `served` temperature,
   `caffeine`, shopper `keywords`, a `blurb`, live `qty`, and the robot `location`
   (turn/lift) copied from the grasp config. These attributes were generated by a model and
   are what the recommender reasons over.

2. **Scenario implementation (model + prompts + recommendation state machine).**
   [`ENGINE.md`](ENGINE.md) — the full spec: a 9-state / 19-transition state machine, the
   need→function inference cues, constraint extraction, a deterministic ranking formula, the
   clarifying-question policy + bank, accept/decline detection, and the **LLM prompt
   templates** (system prompt, reasoning scratchpad, and a JSON action schema
   `ask_clarifying` / `recommend` / `confirm_order`) plus fine-tuning knobs. This is
   implemented in vanilla JS inside the demo.

3. **Website / platform display (interactive demo).**
   [`index.html`](index.html) — a self-contained page (no build, no network). Open it and chat.
   It shows three live views: the **chatbot**, the **reasoning scratchpad** (need → function →
   constraints → candidates → exclusions → inventory check → decision), and the **store
   inventory** board (highlights matches, rings the pick, dims filtered-out items).

## How it works

**Per need** (one recommendation cycle):
```
utterance ─▶ normalize ─▶ score 5 function tags (keyword cues)
          ─▶ extract constraints, seeded from remembered preferences
          ─▶ if unclear or two categories tie → ask ONE clarifying question (cap 2)
          ─▶ filter: function ∩ constraints ∩ in_stock, minus already-declined
          ─▶ rank: 3·functionMatch + 2·keywordOverlap + sweetnessFit + sugar/temp/stock
          ─▶ recommend top SKU ; decline/new constraint → re-rank, never re-offer
```

**Across the conversation** (the multi-turn layer):
```
GREETING → ELICIT_NEED → [need cycle] → RECOMMEND
   accept(+qty) → add to cart → POST_ADD ("anything else?")
        ├─ new need  → ELICIT_NEED (preferences carried over)
        ├─ remove X  → drop from cart
        └─ checkout  → cart summary + robot pick-plan → reopen for more
```
Conversation memory (`profile`, `cart`, `lastOptions`) lives on the session and survives each
order — that's what makes it multi-turn rather than one-shot.

## The primary engine — offline, rule-based, LLM-*like*

The rule engine answers **first, on every turn** — instantly and deterministically (it's also
what the automated tests exercise). It's tuned to *feel* like an LLM — it copes with messy,
natural phrasing instead of exact keywords:

- **Stemming & typo tolerance** — *"just smashed a gym session"*, *"electrolite water"*,
  *"gimme somethin fruty"* all resolve correctly (edit-distance matching, restricted to longer
  words so it never confuses *sweet* with *sweat*).
- **Negation** — *"no juice, just water"*, *"not a gift, something to drink"* drop the negated
  category and keep the rest.
- **Small talk & open-ended** — *"hi"*, *"thanks"*, *"what do you have?"*, and *"surprise me"*
  are all handled (the last one just makes a solid pick).
- **Function-aware ranking** — plain thirst → water; *"fruity"* → juice; *"treat"* → ice cream;
  post-workout → the low-sugar electrolyte.
- **Varied, warm replies** — phrasing rotates so it doesn't read like a script.

It still asks a clarifying question when a request is genuinely ambiguous (*"something nice"*),
and keeps all the multi-turn behaviour (cart, memory, references, checkout). No API key, no
account, no cost — open the file and go.

## The backstop engine — an in-browser LLM (WebLLM · WebGPU)

When a message falls outside what the rules recognise, **that turn** is handed to a small language
model that runs **entirely in the visitor's browser** via WebGPU ([WebLLM](https://github.com/mlc-ai/web-llm)) —
no server, no API key, no per-message cost, nothing to install. It runs the full action set
(recommend / add_to_cart / checkout), streams its reply, and syncs the rule engine's state so the
**next** turn is instant again.

How it works: on the **first** escalation the page lazily `import()`s WebLLM from a CDN and downloads
the model (**`Qwen2.5-1.5B-Instruct`, ~1 GB**, one bilingual model), showing a progress % in the
chat bubble; the browser caches the weights so later visits are instant. Each escalated turn sends
the system prompt + compact catalog + conversation + live cart, and the model returns a JSON action
that JS validates (SKU must exist, qty capped to stock). **If the browser lacks WebGPU or the model
fails to load, that turn falls back to a single clarifying question** — so the file works for everyone,
the rule engine always answers, and only the rare unrecognized turn needs the model.

Swap the model in `index.html` (`WEBLLM_MODEL`): smaller/faster → `Qwen2.5-0.5B-Instruct-q4f16_1-MLC`;
higher quality → `Llama-3.2-3B-Instruct-q4f16_1-MLC`.

## Sharing it — send one file to everyone

You just send **`index.html`** (self-contained; no build step). What each recipient gets:

- **The rule engine works for everyone, always** — offline, no setup, no GPU, instant.
- **The in-browser AI** works in any **WebGPU** browser (Chrome/Edge, Safari 17+) on a reasonably
  capable machine; the ~1 GB model downloads on the first unrecognized message and is cached after.
  On unsupported browsers the demo simply stays on the rule engine.
- **Most reliable over http(s)**: opening a double-clicked `file://` usually works, but WebGPU +
  CDN fetches are most dependable when the page is served — drop `index.html` on any static host
  (GitHub Pages, Netlify, Cloudflare Pages, or `python3 -m http.server`) and share the link.
- **No secrets to leak**: because the model runs client-side, there's no API key embedded in the
  file for anyone to extract — unlike a cloud-API build, which you should never ship as a raw file.

**Latency notes** (in-browser model speed = model size + the visitor's GPU):

- **Fewest calls** — rule-first means the model is only invoked on genuinely unrecognized input.
- **Small prompt** — the catalog is sent as a compact one-line-per-item table (no `keywords`/blurb),
  ~66% fewer tokens than JSON, so it prefills faster.
- **Bounded output** — `max_tokens: 256` (the reply is one small JSON action) so it stops quickly.
- **Streaming** — the reply types out live (time-to-first-token) while the model is still generating.
- **Cached after first load** — the weights persist in the browser, so only the first-ever escalation
  pays the download; the model then stays resident for the session.

## Bilingual — English / 中文

Toggle **EN / 中文** in the chat header. It switches the whole UI, the bot's understanding,
and its replies. In Chinese it handles the same behaviours — *"刚健身完，不要太甜"* → the
low-sugar electrolyte water, *"不要果汁，我要水"* (negation), *"随便推荐一个"* (surprise),
*"来两个"* (quantity), *"换粉色那个"* (reference), *"结账"* (checkout with a 中文 pick-plan) —
and each mode still understands the other language too.

## Tests

```
node test.js        # 68 checks; exit 0 = all pass
```

`test.js` loads `index.html`, extracts the (DOM-free) rule engine, and drives real scripted
conversations through it — asserting on both the returned action and the mutated session
(state / cart / remembered preferences). Coverage:

- **Multi-turn** — workout → recommend → *"make it two"* → ice cream → multi-item checkout;
  preference memory carries across needs; *"the pink one instead"*; options + *"the second
  one"*; *"too sweet"* refine; negation (*"no juice, just water"*).
- **Hybrid dispatch** — recognized turns stay on the rule engine (instant, no escalation);
  genuinely unrecognized input (*"tell me a joke"*, *"do you accept credit cards"*) returns
  `escalate` so the LLM takes over; small talk / *"surprise me"* / *"not sweet"* stay on the rules.
- **中文** — Chinese phrasing → right category, quantity (*"来两瓶"*), graceful empty checkout,
  and escalation on unrecognized input.
- **Product Q&A** — *"is it cold?"*, *"how much sugar?"*, *"does it have caffeine?"*, *"how many
  calories?"*, *"what's in it?"* are answered **precisely** and keep the same pick (EN + 中文).
- **Regressions for the fixes below** — checkout from any state, empty-cart checkout, *"you
  pick"* in mid-cart, the middle sweetness tier, *"one more"* re-add, browse-vs-checkout.

Recent fixes (all covered by the suite):

- **Checkout from any state** — the always-enabled Checkout button previously re-offered the item
  when a recommendation was on screen; *"that's it" / "I'm done"* now check out (with the pick-plan)
  instead of abandoning and clearing the cart; empty-cart checkout replies gracefully.
- **Precise product answers** — a question like *"is it cold?"* is answered directly and keeps the
  current pick, instead of being misread as a *"make it cold"* refinement that re-ranks.
- **`"one more"` / `"two more"`** re-adds the item you just added (was escalating to the LLM).
- **Browse ≠ checkout** — *"something else" / "what else do you have?"* now show the menu and stay
  open for another item, instead of checking out and clearing the cart.
- **`"you pick"` / `"whatever"`** after adding an item recommends instead of a redundant clarifier.
- **Middle sweetness tier** — *"a little sweet"* targets a ceiling of 3 rather than the sweetest item.
- **Honest about caffeine** — nothing in the catalog is caffeinated, so *"anything with caffeine" /
  "wake me up" / "coffee"* now get a truthful "we don't stock that" reply (with an alternative) instead
  of a caffeine-free juice passed off as one. Also fixed *"anything with X"* being misread as *"surprise me"*.
